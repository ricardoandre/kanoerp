internal production-management system on NocoBase. **NocoBase is the live source
of truth** (it executes the code); this repo is the history/context mirror and
the onboarding doc for any new Claude session.

## 1. What this project is

Internal garment production management system, built as inline JavaScript pasted
directly into **NocoBase** (self-hosted, Docker on Ubuntu `ubuntu-olin`,
accessible at `app.kano.id:13000`, Jakarta UTC+7 timezone, MySQL backend,
`DD/MM/YYYY` date formatting). No filesystem deployment for the core app — every
"file" here corresponds to one NocoBase row (`source_code`, `jblock`, or
`JSAction`).

Tracks productions across two internal brands (product code prefix `A` vs `O`)
through a full status workflow: `planning → cutting → production → QC → permak →
done`. Core entities: productions, konveksi (external production partners),
products/SKU variants, materials, samples, delivery/QC results, permakan
(alteration/rework), and retail outlet stock.

Separate components with their own deployment models:
- **Facebook Ads → NocoBase pipeline** (`~/fb-ads-sync`, Node.js) — see §10.
- **Instagram insights pipeline** — schema finalized, not yet live — see §13.2.
- **NocoBase custom plugin path** — a genuine filesystem-deployed plugin, distinct
  from the inline-JS rows that make up the rest of this app — see §13.6.

## 2. Sync workflow

1. Code is generated/edited in a Claude session.
2. Paste into the matching NocoBase row (runs immediately — NocoBase is live).
3. Commit the same change here (paste-and-commit, no delete/reupload).
4. Next session: get current file(s) before editing — see the callout at the top.

Clone (if Claude has bash+network, or for Andre to run and paste the output):
```
git clone --depth 1 https://github.com/ricardoandre/kanoerp.git
find kanoerp -type f -not -path '*/.git/*'
```
GitHub's web "tree" page blocks automated fetching (`robots.txt`), so that's not
an option — the clone above (or asking Andre to run it) is the only reliable way
to get a current file list.

**How code goes live:** `jblock` → new NocoBase Page, one custom-code block,
paste in (one page per jblock). `JSAction` → attached to a native table block's
toolbar button (not a custom jblock). A `source_code` row is never placed on a
page directly — only loaded via `loadCode(name)` from a `jblock`/`JSAction` (see
§3 for the exact signature actually in use).

**Session close ritual:** end every session with an explicit "changed this
session" list — which rows to paste into NocoBase, which files to commit here.
**Then confirm the commit actually happened before the next session assumes it
did** (see the top callout — a drafted-but-unpasted README is exactly the kind of
drift this is meant to prevent).

**Schema refresh ritual:** schema lives in a separate file (§7), regenerated by
the `view_database_schema_dump` jblock. Never hand-edit it from memory — relation
field names aren't predictable from the collection name (§4.1).

## 3. Hard sandbox constraints — never violate these

- **`window.*` fully blocked** (`print`, `open`, `innerWidth`, `addEventListener`,
  `location`, etc.). Never propose a `window.*` fix.
- **`document` only works where already proven** — the one confirmed pattern:
  `document.createElement('a')` + `Blob` + `.click()` for CSV/PDF download.
  **`document.body` and any DOM-insertion (`appendChild`/`removeChild`) is
  blocked** — confirmed broken: "Access to document property 'body' is not
  allowed." `.click()` works standalone, without ever inserting the element into
  the DOM.
- **`fetch()` is blocked; writing to any global throws (SES lockdown)** — no
  external libraries can be loaded (e.g. no binary `.xlsx` parser). CSV only;
  users convert `.xlsx`→CSV before upload.
- **`new FileReader()` is blocked**, but **`Blob.prototype.text()` on an existing
  `File`** (e.g. from antd `Upload`) works fine — it's an instance method, not a
  global construction. Use this to read uploaded file contents.
- **`setTimeout` is blocked.**
- **PDF generation must not use `window.print`** — build PDF bytes by hand in a
  `source_code` row instead (see `ui_prepare_fabric`).
- **Jblocks cannot import each other.** Shared logic lives in `source_code` rows,
  compiled via `new Function('React','antd','dayjs','ctx', src)` and loaded with a
  `loadCode` helper each caller defines locally (not itself shared). **Confirmed
  working signature: `loadCode(name)`** — no `ctx` param, closes over the caller's
  own top-level `ctx`. **This snippet was previously stale in this README** — the
  current version uses `ctx.api.resource('source_code')`, not raw SQL, because of
  the admin-gating bug below:
  ```js
  const _codeCache = {};
  async function loadCode(name) {
    if (_codeCache[name]) return _codeCache[name];
    const rows = await ctx.api.resource('source_code').list({
      filter: { name }, fields: ['code'], pageSize: 1
    }).then(r => r?.data?.data || r?.data || []);
    const src = (rows && rows[0] && rows[0].code) || '';
    _codeCache[name] = new Function('React', 'antd', 'dayjs', 'ctx', src)(React, antd, dayjs, ctx);
    return _codeCache[name];
  }
  ```
  A `source_code` row's content is a function **body** (not a declaration), ending
  in `return {...}` of its exports. Any exported function needing SQL should take
  `ctx` as its own explicit parameter (e.g. `runSql(ctx, prefix, sql)`) rather than
  relying on the outer closure — the cached module could be invoked by a different
  caller/ctx later. `dayjs`/`React`/`antd` are safe to close over directly.
- **Concurrent dynamic-uid `ctx.sql` calls can collide** (`"invalid sql schema uid
  used"`). Fix: one combined query (e.g. `UNION ALL`) under a fixed uid, or
  serialize instead of `Promise.all`-ing several dynamic uids. Multiple *fixed*,
  *different* uids run concurrently is fine.
- **`ctx.sql.save()` is gated to admin/root — non-admin roles get a silent,
  record-by-record failure, not a clean permission error.** Discovered 2026-07
  across `ui_production_material_detail.js` (Details/Quantity/Track tabs) and
  `ui_material_out.js`.

  **Mechanism:** `ctx.sql.save({uid, sql, ...})` *registers* a query under a
  `uid`; `ctx.sql.runById(uid, ...)` *executes* whatever's currently registered
  there. `save()` silently fails for non-admin roles (matches NocoBase's own
  stance that raw SQL execution is admin-only). Once *any* admin/root session
  opens a given record and its `save()` succeeds, that `uid` is registered
  server-side permanently — after that, `runById(uid)` succeeds for *any* role,
  because the query already exists.

  **Symptom pattern that makes this hard to diagnose:** it does NOT look like a
  permission wall. It looks like a flaky, record-specific bug:
  - Fails inconsistently — works on some production records, not others.
  - The dividing line is invisible: it's simply "has an admin opened this exact
    record before" — nothing about the record itself differs.
  - Admin/root testing alone will **never** reproduce it — you need a second,
    non-admin session on a record admin has never touched.
  - `runSql`'s standard `.catch(() => {})` / `.catch(() => [])` (§4.2) makes it
    worse — the real error is swallowed, so it surfaces later as a generic
    `Cannot read properties of null (reading 'sql')` or an empty-result fallback
    ("No production record found") with no indication of *why*.

  **Diagnostic test, if this resurfaces:** two browser sessions, one admin, one
  target role, both incognito/separate profiles. Non-admin opens a *record
  admin has never opened* → reproduce failure. Admin opens the same record once
  → non-admin retries the same record → now succeeds. If that pattern holds,
  this is the bug, not a race condition or a collection-permission gap (both of
  which were incorrectly suspected first — see below).

  **False leads to rule out first, in order, before landing on this:**
  1. Collection-level CRUD permissions (Roles & Permissions → collection →
     Configure permissions) — governs `ctx.api.resource()`, not `ctx.sql`.
     Granting "All records" here does nothing for this bug.
  2. `Promise.all`-ing multiple `runSql` calls — a real, separate bug (§3's
     existing "concurrent dynamic-uid" bullet), but produces a *different*
     symptom: intermittent failure regardless of role, not "always admin,
     record-dependent for everyone else." Fixed independently by serializing
     the loop — worth keeping fixed, but doesn't resolve this issue.
  3. Session/uid collision between concurrent users on the same record — ruled
     out because the fix (per-session uid tagging) wouldn't explain "works
     forever once *any* admin has opened it."

  **The only real fix: stop depending on `ctx.sql` for anything a non-admin
  role needs to run.** Convert the query to `ctx.api.resource(collection).
  get({filterByTk})` / `.list({filter, fields, sort, pageSize})`, which is
  gated by ordinary collection View permissions, not the admin-only SQL
  execution path. For joins/aggregates the raw SQL used to do (JOIN, GROUP BY,
  SUM), fetch each base collection's rows via a plain `.list()` call, batch-fetch
  referenced-id lookups with `{$in: [...]}` (never loop `.get()` per row — that's
  an N+1 anti-pattern), and do the join/aggregation in JS. More verbose than one
  SQL statement, but keeps the feature usable by every role with the right
  collection permissions, no admin "warm-up" step required.

  **Converted so far (2026-07):** `ui_production_material_detail.js` (all three
  tabs) and `loadCode`'s own internal `source_code` lookup (used in every
  `act_*.js` shell — see the fixed snippet above).

  **Correction (2026-07-17) — `ui_material_out.js` was NOT actually converted,
  despite previously being listed here.** The live/mirrored file as of today was
  still on the raw-SQL `runSql`, with `ctx.sql.save()` gated behind
  `ctx.flowSettingsEnabled`. Symptom: clicking "Material Out" from the native
  `production_material` table (JSAction shell `act_material_out`) threw
  `Cannot read properties of null (reading 'sql')` — consistent with this bug's
  mechanism: the dynamic per-record uid (`pm_out_lookup_<id>`) was never
  registered because the gated `.save()` never fired, so `runById` executed
  against an unregistered uid.

  **Interim fix applied today:** removed the `ctx.flowSettingsEnabled` gate so
  `.save()` always fires unconditionally (matching `loadCode`'s pattern).
  **Confirmed this resolves the crash for the admin/root session it was tested
  on.** This is *not* the real fix, though — it just makes `.save()` actually
  attempt registration every time; it does nothing for the underlying
  admin-only-SQL-execution gate. `ui_material_out.js` should be treated as
  **still unconverted** and moved back into the list below — the proper fix
  (rewrite `fetchSummary`/`runSql` in that file to use `ctx.api.resource()`
  instead of `ctx.sql`, per the "only real fix" above) is still pending, and a
  non-admin user opening a `production_material` record for the first time via
  Material Out will likely still hit the silent-failure version of this bug
  until that conversion happens.

  **NOT yet audited/converted — assume broken for non-admin roles until
  checked:** `ui_material_out.js` (see correction above), `ui_production_edit.js`,
  `ui_prepare_fabric.js`, `ui_list_engine.js`, `ui_record_nav.js`, all three
  importers (`ui_import_material_details`, `ui_import_product_main_material`,
  `ui_import_product_material`), `view_production_result`, `fn_fbads_data`
  (and everything built on it — all 3 FB Ads jblocks). **Also unaudited: every
  §13 workstream** (bulk image upload, measurement, product detail view,
  `material_out_request`) since they're newer than this bug's discovery — assume
  broken for non-admin roles and check with the diagnostic above before trusting
  any of them in production with non-admin users.

## 4. NocoBase API patterns

- **MySQL.** Backtick reserved words, `LIMIT n`, no `information_schema` for
  relations (see below).
- **SQL:** `ctx.sql.save({ uid, sql, dataSourceKey: 'main' })` (awaited) →
  `ctx.sql.runById(uid, { type: 'selectRows', dataSourceKey: 'main' })`.
  - Both calls take an **object**, not positional args. `dataSourceKey: 'main'`
    is required on both — omitting it fails or errors.
  - `uid` must be a **fixed string** in column/list contexts; **dynamic per row**
    (`"prefix_" + record_id`) in multi-row column code, to prevent concurrent
    collisions between rows rendering at once.
  - Canonical wrapper: see `fn_fbads_data.runSql(ctx, prefix, sql)` — generates a
    unique uid per call and chains `.save().then(() => .runById())`. Reuse this
    wrapper shape for any new SQL-touching code. **But prefer `ctx.api.resource`
    over `ctx.sql` entirely for anything a non-admin role needs — see §3's
    admin-gating bug before reaching for raw SQL in new code.**
- **Writes:** `ctx.api.resource('collection').create({values:{...}})` — reliable,
  avoids raw-SQL-`INSERT` snowflake-ID issues.
- **`belongsTo` needs nested payloads:** `{ [relFieldName]: { [targetKey]: value } }`
  — a raw FK value alone fails validation. Relation field names aren't
  predictable from the collection name — check the schema file (§4.1, §7).
- **Product table uses `code` as its natural PK (not `id`)** — filter/lookup by
  `code`, not `id`, for anything touching `product`.
- **Image/attachment fields** are relations through obfuscated junction tables —
  use `appends:['image']` via the resource API, or JOIN to attachments via SQL.
- **Enums:** resolve via `ctx.dataSourceManager.getDataSource('main').
  getCollection(...).getField(...).enum` — not raw SQL.
- **Schema introspection:** use `ctx.dataSourceManager.getDataSource('main').
  getCollections()` (`.name`/`.title`/`.template`/`.filterTargetKey`/
  `.getFields()`), not `information_schema` — most relations here are enforced at
  the app layer, not real DB FKs, so a raw FK dump comes back mostly empty.
- **CPAS/Shopee accounts:** conversions only in `catalog_segment_actions`/
  `catalog_segment_value` — standard `actions`/`action_values` silently drops
  them; ad-level (`ads_insights`) is the reliable source. Scoped to conversion
  **metrics** only — CPAS objective metadata comes through normally.
- **Reach/frequency are NOT additive** — not across days, not across entities, at
  any level. Only a period-scoped fetch (no `time_increment`) or a genuinely
  single entity/single day row is correct. No daily-grain deduplicated reach
  source exists for any entity today — see §10.

### 4.1 Relation field names aren't predictable

A `belongsTo`/`hasMany` field's name routinely differs from its target
collection's name or FK column name. Some tables carry legacy duplicate raw
audit columns alongside the standard `createdBy`/`updatedBy` belongsTo pair —
prefer the belongsTo pair. Always check the schema file (§7) before writing a
nested payload or `appends` — never guess or hand-recall from memory.

### 4.2 Canonical `runSql`/`execSql` template

```js
async function runSql(ctx, uid, sql) {
  if (ctx.flowSettingsEnabled) {
    await ctx.sql.save({ uid, sql, dataSourceKey: 'main' }).catch(() => {});
  }
  return ctx.sql.runById(uid, { type: 'selectRows', dataSourceKey: 'main' })
    .then(r => r || []).catch(() => []);
}
```
(`execSql` is the same shape with `type: 'exec'`.) `.save()` only runs when
`ctx.flowSettingsEnabled` is true (likely flow-config-UI vs. live page, exact
reason not fully understood — copy the pattern anyway). Both calls swallow
errors. **Remember: this whole pattern is admin-gated (§3) — use it only for
admin-only tooling (dev tools, schema dump) or reads that genuinely need raw
SQL; use `ctx.api.resource` for anything a normal production/warehouse role
needs to run.** **2026-07-17 confirmed case:** `ui_material_out.js` had the
`.save()` call gated behind `ctx.flowSettingsEnabled` and hit `Cannot read
properties of null (reading 'sql')` when triggered from a JSAction on a native
table — see §3's correction for the full mechanism and why the real fix there
is still pending.

## 5. Architecture principles

- **Extract-on-second-use.** Write inline first; promote to `source_code` only on
  genuine 2nd+ use. (`fn_fbads_data`/`ui_fbads_controls`, §10, were extracted on
  their 3rd shared use. `MEASURE_FIELDS`, §13.3, is a known candidate — currently
  duplicated across three files, not yet extracted.)
- **Downward-only deps:** `ui_` → `fn_` → `lib_`. Never sideways/circular.
- **Stable contracts per module** — one-line shape comment at the top. Extend a
  shared component's contract deliberately rather than forking a copy.
- **Single canonical version** — edit shared modules in place.
- **Rename in place, never delete-and-recreate jblocks** — block UIDs feed filter
  controllers; recreating breaks them silently.
- **Jblock names ≠ importable strings** — only `source_code` rows load by name
  via `loadCode`.
- **Naming:** `ui_`/`fn_`/`lib_` → `source_code`; `view_` → `jblock`; `act_` →
  `JSAction`.
- **`ctx` always threaded as a parameter, never captured at compile time.**
- **Change scope discipline:** full-file replacement for complex changes,
  targeted find/replace for small ones; confirm direction first when ambiguous.
  **On iteration, deliver a complete drop-in replacement of the specific named
  component/section**, not a full-file rewrite of unrelated things.
- **Mockup before implementation** for new UI layout.
- **Preview before writing, opt-in before overwriting** — any bulk-write/import
  feature classifies rows (new/no-op/conflict) and previews before committing;
  overwriting existing non-empty data needs explicit opt-in. Never silent.
- **No N+1 fetches** — batch via `$in` filters; sequential awaits instead of
  `Promise.all` for anything touching dynamic SQL uids (§3).
- **Don't guess unseen files/schemas/env vars — ask or fetch instead.**

## 6. UI conventions

**antd** ([Ant Design](https://ant.design)) via `ctx.libs.antd` — version not
confirmed, so mirror components already proven in this codebase rather than
assume an API. Confirmed working: `Modal` (`.confirm()`, `.destroyAll()`),
`Drawer`, `Dropdown`, `Select`, `DatePicker`, `InputNumber`, `Switch`, `Spin`,
`message`, `Pagination`, `Upload`, `Button`, `Table`, `Alert`, `Progress`,
`Input`, `Tag`, `Segmented`, `Checkbox`, `Popover`.

- `React.createElement` aliased `ce` — **never JSX** (no transpiler).
- Scoped `<style>` tags, not global CSS.
- Scrollable popups/drawers need a trailing ~80–120px spacer div.
- Scrollable overlays: `overflow-y: scroll` + `-webkit-overflow-scrolling: touch`;
  flex children need `min-height: 0`.
- `border-collapse: collapse` silently breaks `position: sticky` in
  Safari/WebKit — use `border-collapse: separate` + `border-spacing: 0`.
- `position: fixed` modals can be clipped by NocoBase's block container — use a
  DOM portal (`document.body.appendChild`) to escape stacking contexts. **(Note:
  this contradicts §3's "`document.body` is blocked" finding — that block was
  confirmed in a different context. If this pattern is reused, re-verify
  `document.body.appendChild` actually works where it's being added; don't
  assume it's safe everywhere just because it worked once.)**
- **Mobile detail pattern:** single expandable bottom sheet, always-visible main
  details + single-open accordion sub-sections. `Edit`+`•••` top-right. No
  separate desktop page yet. NocoBase blocks UI Editor/design mode on mobile
  viewports by design — expected, not a bug.
- **Cross-record nav replaces, never stacks** — via `ui_record_nav`.
- Cross-links: `›` chevron, indigo `#4338ca`, no underline.
- **Status colors:** planning `#f97316` · cutting/production `#d97706` · QC
  `#84cc16` · permak `#ef4444` · done `#22c55e`. Orange = pending, green = done,
  zero qty = orange.
- **FB Ads reports (§10):** objective badges — purchase `#16a34a`, atc `#f59e0b`,
  reach `#3b82f6` (also covers merged traffic objectives), other `#6b7280`.
  Dashed "guessed" badge = no synced `objective_key` yet; amber "⚠ mixed" =
  adsets disagree on `custom_event_type`. Both informational only. Missing real
  data → "—" + warning banner, never a silent approximation.

**Two known patterns — only one correct:** DETAILS CODE (`ctx.libs.React`, async
`bootstrap()` in `useEffect`, `runSql()`, `ctx.render()`) is correct for any
detail/popup. COLUMN CODE (`ctx.element.innerHTML`, chained `.then()`, no React)
is broken for detail pages — only use it for simple list-column cells.

## 10. Facebook Ads → NocoBase pipeline

`~/fb-ads-sync` — production path is `~/github-nocobase/kanoerp/fb-ads-sync`
(**not** `/root/kanoerp` — a stray clone there should be removed if it still
exists). Several independent scripts sharing account config + a NocoBase client.
Real file listing (confirmed via `ls`): `sync.js`, `sync-today.js`,
`sync-periodic.js`, `backfill.js`, `backfill-periodic.js`, `creatives.js`,
`fetch-status.js`, `lib/{accounts,creatives,facebook,nocobase,transform,
transform-period}.js`.

- **`lib/accounts.js`** — `parseAccounts()` reads `AD_ACCOUNTS`
  (`Label:act_xxx,Label2:act_yyy`, bare `act_xxx` OK too), falls back to legacy
  single `AD_ACCOUNT_ID`. `filterAccountsFromArgs(accounts, argv)` supports
  `--account=Label1,Label2`. **Every Graph API script must loop
  `parseAccounts()` itself** — `lib/facebook.js`'s `fetchInsights`/
  `fetchEntityStatus` take `accountId` per call, no internal looping.
- **`lib/facebook.js`** — `fetchInsights({accountId, level, since, until,
  timeIncrement})` (`timeIncrement:1`=daily row, `null`=one aggregated row for
  periodic scripts) and `fetchEntityStatus({accountId, level})` (object
  endpoints `/campaigns` `/adsets` `/ads`, not `/insights`, used by
  `fetch-status.js`). `LEVEL_FIELDS` includes `'account'`.
- **`lib/nocobase.js`** — `upsert`/`upsertMany` (`:updateOrCreate?filterKeys[]=`),
  `getOne` (`:list?filter=`), `uploadAttachment`.
- Run scripts from the project root. `pickConv`/`pickRevenue`: CPAS arrays first,
  pixel fallback. Backfill: `BACKFILL_SINCE=2024-01-01 node backfill.js`.
- **Daily account-level reach/frequency**: `--period=day` added to
  `sync-periodic.js` and `backfill-periodic.js`. **Crontab line still needed** —
  `node sync-periodic.js --period=day` at `0 1 * * *` — confirm this has
  actually been added before assuming it's running.
- **`creatives.js` gotchas:** "reduce the amount of data" errors aren't
  transient — `lib/creatives.js` halves `limit` in place (50→floor 5) rather
  than retrying blind, and paginates manually via cursor so the shrunk size
  sticks. Code 17 (account throttle) is a sliding window, not a blip — fails
  fast + moves to the next account rather than backing off.
- **Known open issue:** `ASKALABEL_WEB` was still hitting code-17 on the last
  test even with the fail-fast fix deployed — needs the account to cool down;
  if recurring across sessions, check for another concurrent caller of that
  account's API (quota is account-wide).
- **`lib/nocobase.js` write path unverified end-to-end** — every test run failed
  during the fetch step before reaching writes. Don't assume `ads_creative`'s
  schema or `getOne`'s filter shorthand are correct until a run gets past fetch.
- **Secrets: `.env` is gitignored, not committed.** This repo is public. Real
  credentials must never be committed — if `.env` was ever accidentally
  committed, the fix is rotating credentials immediately, not just deleting the
  file (git history keeps old commits retrievable). `node_modules/`/`*.log` also
  gitignored.

### FB Ads objective classification (real data, not name-guessing)

`fetch-status.js` writes `fb_ads_status.objective_key` on campaign rows:
```
objective:optimization_goal:custom_event_type
```
`objective` = raw `campaign.objective`. `optimization_goal`/`custom_event_type`
= distinct values across the campaign's **adsets**, comma-joined if they
disagree (the split only shows at the adset level).

Read side: `fn_fbads_data.resolveObjective(objectiveKey, campaignName)`:
- `REACH_OBJECTIVES = ['OUTCOME_AWARENESS','REACH','BRAND_AWARENESS',
  'OUTCOME_TRAFFIC','LINK_CLICKS']` → `'reach'`. Traffic merged in deliberately.
- `SALES_OBJECTIVES = ['OUTCOME_SALES','CONVERSIONS','PRODUCT_CATALOG_SALES']` →
  adsets' `custom_event_type`: all `ADD_TO_CART`→`'atc'`, all `PURCHASE`→
  `'purchase'`, mixed→best-effort (prefers `PURCHASE`, then `ADD_TO_CART`, else
  `'other'`), no event data→name-guess fallback.
- Anything else → `'other'`.
- Legacy pre-ODAX strings confirmed still occur on older campaigns — handled.
- `isGuessed`/`isConsistent` flags surface as small non-blocking badges — never
  block rendering.

### Live targets (`fb_ads_target`)

`cost_atc` (<1200, guards `atc` count), `cost_reach` (<16, guards `reach`),
`roas` (>15, guards `spend`), `cost_purchase` (<20000, guards `purchase`).
`countField` stops a 0-from-no-data value reading as if it "beat" a `<` target —
not itself overridable, only `target`/`direction` are.

### `view_fb_report_overview` redesign

One single account-total view, no tabs, no objective classification — a
"bucket" isn't a real Facebook entity, so its reach could never be more than a
sum of per-campaign numbers, never truly deduplicated. Spend/Link
Clicks/ATC/Purchase/Revenue: exact `SUM(...)`, genuinely additive.
Reach/Impressions/Frequency: only from `fb_ads_period_data`
(`entity_type='account'`), Weekly/Monthly only. Missing period → "—" + warning
banner, never derived. **Daily removed entirely** at the account level (no real
daily-grain source exists — see the crontab item above, which is trying to fix
this).

### Shared code + account filtering

`fn_fbads_data`/`ui_fbads_controls` extracted on 3rd shared use. All 3 FB Ads
reports are single-select Account — empty selection = all accounts combined.

## 11. Working style

- Modular delivery: shared helpers → data-fetching → small presentational
  sub-components (`createElement`) → thin composition root.
- Complete drop-in replacement of the **specific named component** on iteration —
  not a full-file rewrite of unrelated things.
- One concrete step at a time, confirm before moving on — don't assume
  unconfirmed prerequisites.
- End-of-session: remind Andre to (a) paste finalized code into the matching
  NocoBase row, (b) commit here, (c) update this README if anything changed —
  and **confirm the commit actually landed**, don't assume it did.
- Fetch the schema file (§7) before SQL/`belongsTo` against an unfamiliar table.
- **Claude cannot execute this code against live NocoBase** — the paste→report-
  errors-back loop *is* the verification step, not a sign something's wrong.
- Confirm field names via introspection or the schema file rather than guessing.
- Read relevant code files before editing — never assume structure or schema.
- For multi-file/multi-decision work, confirm scope and architectural decisions
  before writing code whenever there's more than one reasonable interpretation.
- Syntax-check with `node --check` using correct wrapper signatures:
  `source_code` rows use 4-param `new Function('React','antd','dayjs','ctx', src)`;
  jblocks use `ctx`-only or `async ctx`-only wrapper.
- All files delivered with `.js` extension so they render in the side panel for
  copy-paste.
